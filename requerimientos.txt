Introduccion

Este documento describe los requerimientos funcionales y técnicos de un sistema ERP de
tickets, basado en Angular 21 + PrimeNG, con autenticación por usuario, permisos por
acción (sin roles) y arquitectura de microservicios con un API Gateway.
El objetivo es permitir a los usuarios gestionar tickets en modo Kanban o lista dentro de
grupos / workspaces, con controles de seguridad finos a nivel de interfaz y de API.


Stack Tecnologico
Frontend
• Framework: Angular 21 con enrutamiento estándar (@angular/router).
• Librería de UI: PrimeNG
• Estructura de componentes:
• Cada componente vive en una carpeta separada con:
• component.ts (lógica).
• component.html (vista).
• component.css (estilos locales).

Backend
• Arquitectura de microservicios:
• user (gestión de usuarios, login y register), tecnologia: UP2U.
• tickets (gestión de tickets), tecnologia: fastify.
• groups (gestión de grupos / workspaces y permisos por grupo), tecnologia:
UP2U.
• apigateway (punto único de entrada para todas las llamadas, validación de
token y permisos), tecnologia: fastify.

• Todos los requests pasan obligatoriamente por el API Gateway.
• Comunicación entre servicios mediante HTTP/REST.
Modelo de seguridad y permisos
Permisos por acción (n ROLES)
• El sistema se basa en permisos expresados como strings (por
ejemplo: "tickets:add", "tickets:move", "groups:manage", "users:manage").
• los permisos se asignan directamente por usuario y grupo.
• Cada usuario puede tener permisos diferentes por grupo / workspace.
Frontend: directiva de permisos
• Crear una directiva Angular reutilizable llamada appHasPermission, esta se utilisa
en el renderizado en interfaz y/o rutas.
• Funcionamiento:

• Si el usuario tiene el permiso especificado, el elemento se renderiza.
• Si no lo tiene, el elemento se elimina del DOM (no solo ocultado con CSS).
• La directiva debe consultar un servicio central (PermissionService) que exponga:
• hasPermission(permission: string): boolean.
• refreshPermissionsForGroup(groupId: string): void.

Backend: permisos en el API Gateway
• El API Gateway debe:
• Validar el token de autenticación (JWT o equivalente).
• Verificar que el usuario tenga el permiso necesario para el endpoint
solicitado.
• Ejemplo:
• Endpoint POST /tickets requiere permiso "tickets:add".
• Endpoint PATCH /tickets/:id/status requiere
permiso "tickets:move" y que el ticket esté asignado al usuario.

• Si el usuario no tiene el permiso:
• Devolver código HTTP 403 y el mismo esquema JSON global.

Endpoints de login y register
• Los endpoints /auth/login y /auth/register del microservicio user:
• No requieren permisos previos.
• El API Gateway solo valida que el request sea válido y enrutará
directamente al microservicio.

Esquema de respuesta JSON universal
Todos los microservicios y el API Gateway deben devolver respuestas con el mismo
esquema JSON:
json
{
"statusCode": 200,
"intOpCode": "SxUS200",
"data": [ /* lista de objetos o objeto */ ]
}
• statusCode: código HTTP semántico (200, 400, 403, 404, 500, etc.).
• intOpCode: código de operación interno del sistema (ej. "SxUS200" para operaciones
de usuario).
• data: payload de la respuesta (array de objetos, objeto único o null).

Vistas y flujos de usuario
Login	
• Vista: LoginComponent.

• Campos:
• Usuario (username o email).
• Contraseña.

• Flujo:
• Enviar credenciales a /auth/login del microservicio user vía API Gateway.
• Si la respuesta es exitosa:
• Guardar token (en cookies).
• Guardar lista de permisos por grupo.
• Redirigir al Dashboard.

Register
• Vista: RegisterComponent.
• Campos:
• Nombre.
• Email / Username.
• Contraseña (y confirmación).
• Flujo:
• Enviar datos a /auth/register del microservicio user vía API Gateway.
• Si la respuesta es exitosa, redirigir al Login o mostrar mensaje de
confirmación.

Dashboard.
• Vista: DashboardComponent.
• Funcionalidades:
• Mostrar estadísticas generales de tickets (por ejemplo: total, por estado,
por prioridad) usando gráficos o tablas de PrimeNG.
• Listar los grupos / workspaces a los que pertenece el usuario.
• Permitir al usuario seleccionar un grupo (por ejemplo, con un combo o
selector).
• Al seleccionar un grupo, cargar el contexto de tickets de ese grupo.

Kanban / Lista de tickets
• Vista: GroupTicketsComponent (o similar).
• Modos de visualización:

• Kanban:
• Columnas de estado (por ejemplo: “To-Do”, “In Progress”, “Done”).
• Tarjetas de tickets que se pueden arrastrar entre columnas
(drag-and-drop si es necesario).

• Lista:
• Tabla con columnas: ID, Título, Estado, Prioridad, Asignado, Grupo.

• Reglas de negocio:
• El usuario solo puede mover el estado de un ticket si:
• El ticket está asignado a él.
• El usuario tiene el permiso "tickets:move".

• Permisos de UI:
• Botón “Agregar Ticket” solo visible si el usuario tiene "tickets:add".
• Botón “Mover estado” solo visible si el usuario tiene "tickets:move" y el
ticket está asignado a él.

Admin Group
• Vista: AdminGroupComponent.
• Funcionalidades:
• Crear, editar y eliminar grupos / workspaces (si el usuario
tiene "groups:manage").
• Asignar usuarios a grupos.
• Configurar permisos por grupo y por usuario (por ejemplo, definir que el
usuario X tenga "tickets:add" en el grupo A pero no en el grupo B).

Admin User
• Vista: AdminUserComponent.
• Funcionalidades:
• Listado de todos los usuarios del sistema.
• Edición de datos básicos (nombre, email, username).
• Asignación de permisos agregados por grupo (si el usuario
tiene "users:manage").
UX (Experiencia de usuario)
Navegación
• Flujo principal:
• Login → Dashboard → Selección de grupo → Kanban / Lista de tickets.
• Menú lateral (usando Sidebar o PanelMenu de PrimeNG) con opciones:

• Admin (grupos y usuarios, solo si el usuario tiene los permisos).
• Perfil
Kanban
• Drag-and-drop intuitivo entre columnas (usar p-dragdrop de PrimeNG si es
necesario).
• Cada columna muestra:
• Título del estado.
• Lista de tickets asignados a ese estado.

Lista de tickets
• Tabla PrimeNG con controles de:
• Filtros: por estado, por prioridad, por asignado.
• Ordenamiento: por ID, título, prioridad, fecha.
• Paginación.

Estadísticas en Dashboard
• Charts o resúmenes visuales:
• Total de tickets.
• Tickets por estado.
• Tickets por prioridad.
• Tickets por grupo (si el usuario pertenece a varios).

Seguridad y performance (backend)
Autenticación y autorización
• Uso de JWT (o token similar) para autenticación.
• El token debe incluir:
• userId e información base del usuario.
• Lista de permisos por grupo (o un identificador que permita recuperarlos
rápidamente).
Rate limiting
• Implementar rate limiting en el API Gateway:

• Limitar número de requests por IP / usuario en un intervalo de tiempo (por
ejemplo: 100 requests por minuto).
• Si se excede el límite:
• Devolver código HTTP 429.
• Incluir mensaje “Too many requests” en el esquema JSON global.

Validación de entradas
• Cada microservicio debe validar las entradas usando JSON Schema o librería
equivalente.
• Ejemplo de validación:
• Campos obligatorios.
• Formatos de email, tamaño de campos, tipos de datos.

Autenticación

QUIERO poder registrar mi cuenta
PARA poder iniciar sesión en el sistema.
• COMO usuario
QUIERO iniciar sesión con mis credenciales
PARA acceder al Dashboard y a mis grupos / workspaces.
Dashboard y grupos
• COMO usuario autenticado
QUIERO ver mi Dashboard
PARA visualizar estadísticas generales de tickets y mis grupos.
• COMO usuario autenticado
QUIERO seleccionar un grupo / workspace
PARA visualizar y gestionar los tickets de ese grupo.
Gestión de tickets
• COMO usuario
QUIERO ver tickets en modo Kanban o Lista
PARA organizar mi trabajo según mi preferencia.
• COMO usuario asignado a un ticket
QUIERO mover el estado del ticket

SOLO si tengo el permiso "tickets:move"
PARA no cambiar estados sin autorización.
• COMO usuario con permiso "tickets:add"
QUIERO ver el botón “Agregar Ticket”
PARA poder crear nuevos tickets.
Administración de grupos y usuarios
• COMO administrador con permiso "groups:manage"
QUIERO crear, editar y eliminar grupos / workspaces
PARA organizar el trabajo en diferentes espacios de trabajo.
• COMO administrador con permiso "users:manage"
QUIERO asignar permisos por grupo a usuarios
PARA que cada usuario tenga permisos específicos según el grupo.

Puntos Adicionales (extra)
El porcentaje se basa dependiendo a la cantidad de puntos extra asignada en
total.
• (20%)Logs centralizados en el API Gateway y microservicios(almacenarlas en B.D):
• Registro de requests (endpoint, método, usuario, IP, status HTTP).
• Errores con stack traces básicos.
• (20%)Métricas simples (almacenarlas en B.D.):
• Número de requests por endpoint.
• Tiempo de respuesta promedio.
• (60%)Realizar Deploy de toda el sistema.

Observaciones
Por cada observacion o incumplimiento de los requerimientos se reducen 4
puntos. Debe tener a consideracion la Experiencia de usuario, validar la interfaz, no
usar alerts, ademas usar colores, tipograficas, estructura, que favorezcan la
visualizaacion del contenido.
